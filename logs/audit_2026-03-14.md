## BUG 1 — Daily Loss Limit Gate is Dead Code

**Severity**: CRITICAL
**File**: `scripts/live_executor.py:2042`

**What**: `check_daily_loss_limit()` returns `daily_loss_dollars` (a negative number for losses, e.g., `-3000`). The gate compares it against `DAILY_LOSS_LIMIT = 0.02` (a percentage). Since `-3000 >= 0.02` is always `False`, the 2% daily loss gate **never fires**.

```python
# Line 2042 — BROKEN
if check_daily_loss_limit(executor_state, equity) >= DAILY_LOSS_LIMIT:
    return  # This NEVER executes
```

The function internally logs an error at line 301 (`abs(daily_loss_dollars) >= DAILY_LOSS_LIMIT * equity`) but has no enforcement — it just logs and returns the dollar value. The downstream `risk_mgr.check_all()` catches losses at **3%** (`max_daily_loss_pct=0.03`), but the intended **2%** limit is completely dead.

**Impact**: Between 2% and 3% daily loss, trading continues when it should be locked out. On $150K equity, this is $1,500-$4,500 of unprotected exposure.

**Fix**:
```python
daily_loss_dollars = check_daily_loss_limit(executor_state, equity)
if abs(daily_loss_dollars) >= DAILY_LOSS_LIMIT * equity:
    return
```

## BUG 2 — `trade_date` Undefined in Override Path → NameError Crash

**Severity**: HIGH
**File**: `scripts/live_executor.py:3139`

**What**: When the executor processes an override (from light_decision or pattern_scanner), the `trade_date` variable is never assigned. It's only defined inside the `else` branch at line 2413 (full pipeline path). After executing the trade, the code unconditionally reaches:

```python
# Line 3139 — CRASHES on override path
with open(LOG_DIR / f"agent_reports_{trade_date}.json", "w") as f:
    json.dump(reports, f, indent=2)
```

**Impact**: After every override trade, the executor crashes with `NameError: name 'trade_date' not defined`. State IS saved (lines 3125, 3137 execute before 3139), but the lock is released, error traceback is logged, and agent reports aren't written. This could confuse monitoring and leave stale report files.

**Fix**:
```python
# Add near line 2349 (before override check), or at top of main():
trade_date = datetime.now(timezone.utc).strftime("%Y-%m-%d")
```

## BUG 3 — Fund Manager Always Sees 0h Hold Time

**Severity**: HIGH
**File**: `scripts/live_executor.py:2291-2308` + `tradingagents/agents/managers/fund_manager.py:49`

**What**: `portfolio_ctx` sets `age_bars` but never sets `hours_held`. Fund Manager reads `portfolio_context.get("hours_held", 0)`, which always returns `0`.

```python
# live_executor.py:2307 — sets wrong key
portfolio_ctx["age_bars"] = int(hours / 4)

# fund_manager.py:49 — reads the right key (never set)
hours_held = portfolio_context.get("hours_held", 0)
# FM prompt line 82: "Held: 0.0h" — ALWAYS
```

**Impact**: Fund Manager cannot distinguish a 1-hour-old position from a 4-day-old one. Time-based decisions (stale position management, flip-flop prevention context) are broken at the FM level. Could lead to premature exits or holding stale positions.

**Fix**:
```python
# live_executor.py:2307, add:
portfolio_ctx["hours_held"] = hours
```

## BUG 4 — `override_direction_allowed` Has Hardcoded Stale Price Levels

**Severity**: HIGH
**File**: `scripts/live_executor.py:208-209`

**What**: Short-selling validation hardcodes BTC price levels from an old market context:
```python
band_low, band_high = 70273.0, 70400.0
breakdown_level = 69620.0
```

With BTC at ~$80K+, `current_price >= breakdown_level` is always `True`, meaning **all short overrides are permanently blocked** by line 211.

**Impact**: Pattern scanner and light decision layer cannot trigger short entries via override, regardless of market conditions. Only the full agent pipeline can open shorts. Short signals from the fast layers are silently discarded.

**Fix**: Remove the hardcoded levels and use dynamic ATR-based validation, or remove this function entirely since the full pipeline already validates direction:
```python
def override_direction_allowed(override_decision, current_price):
    """Allow all directions — the pipeline validates thesis quality."""
    return True, "override direction accepted"
```

## BUG 5 — Close Position Uses Ticker Price, Not Fill Price

**Severity**: HIGH
**File**: `scripts/live_executor.py:2697, 2754`

**What**: After calling `close_position()` (which returns the order with fill data), the code fetches the current ticker price for PnL calculation instead of using the actual fill price:

```python
close_position(exchange, p)
exit_price = float(exchange.fetch_ticker(SYMBOL)["last"])  # ← NOT the fill price
```

`close_position()` returns the order object at line 862, but the return value is never captured at lines 2696 or 2753.

**Impact**: PnL calculations for closed trades are inaccurate. In volatile markets, the ticker can move significantly between the fill and the next API call, leading to incorrect:
- `pending_reflection` returns (agents learn from wrong data)
- `closed_trades` records
- `close_pnl_pct` in trade records

**Fix**:
```python
order = close_position(exchange, p)
if order:
    exit_price = float(order.get("average") or order.get("price") or exchange.fetch_ticker(SYMBOL)["last"])
```

## BUG 7 — ADX Division by Zero in Flat Markets

**Severity**: MEDIUM
**File**: `tradingagents/dataflows/crypto_technical_brief.py:60`

**What**:
```python
dx = (abs(pdi-mdi)/abs(pdi+mdi))*100
```
When both `pdi` and `mdi` are 0 (perfectly flat market or insufficient data), `abs(pdi+mdi)` = 0, causing `ZeroDivisionError` or `inf/NaN` propagation.

**Impact**: Technical brief generation crashes or produces NaN values that corrupt downstream scoring. This can happen during low-activity periods (weekends, holidays).

**Fix**:
```python
denom = abs(pdi + mdi)
dx = ((abs(pdi - mdi) / denom.replace(0, np.nan)) * 100).fillna(0)
```

## BUG 8 — `manage_trailing_stop` State Save Gap After Position Close

**Severity**: MEDIUM
**File**: `scripts/live_executor.py:1560-1597`

**What**: When a trailing stop hits (lines 1576-1582 for longs, 1590-1597 for shorts), the code calls `close_position()` and removes `active_trade` from state, but does NOT call `save_state()` before returning. The state is only saved much later by the caller.

```python
close_position(exchange, p)
# ... calculate returns ...
state.pop("active_trade", None)
return state  # ← No save_state() call!
```

If the process crashes between the exchange close and the eventual `save_state()` in `main()`, `active_trade` is still in the state file but the position is closed on exchange. The reconciliation logic at lines 2067-2147 would eventually fix this, but in the meantime:

**Impact**: Potential for orphaned TP/SL orders on exchange (not cancelled because active_trade wasn't properly cleared). The reconciliation path does run `_cancel_orphan_conditionals()`, but there's a window where duplicate conditional orders could trigger.

**Fix**: Add `save_state(state)` after `state.pop("active_trade", None)` in both trailing stop hit branches (after lines 1582 and 1597).

## BUG 10 — Light Decision CLOSE Bypasses Report Freshness Check

**Severity**: MEDIUM
**File**: `scripts/light_decision.py:284-305`

**What**: `should_act()` for CLOSE returns `True` after position validation but **skips** the `report_age_hours` freshness check that gates BUY/SELL decisions. A CLOSE decision based on stale reports (>4h old) is not filtered.

```python
if action == "CLOSE":
    # ... validates position exists ...
    return True  # ← Skips freshness, confidence, min_move checks
```

**Impact**: The light layer could close a profitable position based on outdated intelligence. The position validation (direction mismatch check) helps, but stale reports might describe conditions that no longer exist.

```python
if action == "CLOSE":
    if report_age_hours is not None and report_age_hours > MIN_REPORT_FRESHNESS_HOURS * 2:
        log.warning(f"CLOSE rejected: reports too stale ({report_age_hours:.1f}h)")
        return False
    # ... existing position validation ...
```

## BUG 11 — `write_realtime` in price_ws.py Has No Error Handling

**Severity**: LOW
**File**: `scripts/price_ws.py:86-91`

**What**: The atomic write uses `tmp.write_text()` then `tmp.rename()`, but if `write_text()` fails (disk full, permissions), the exception propagates up and could crash the websocket monitor loop (though the outer `try/except` at line 127 catches it).

However, if `tmp.rename()` fails, the `.tmp` file persists and is never cleaned up. Over time, failed writes could accumulate `.tmp` files.

**Impact**: Minimal — the outer loop handles it. But disk-full scenarios could cascade.

## BUG 12 — `_signal_handler` Calls `save_state` Inside Signal Handler

**Severity**: LOW
**File**: `scripts/live_executor.py:99-108`

**What**: Despite the comment at line 83 saying "ONLY set flag and log. No I/O inside signal handlers", lines 99-108 call `save_state()` which performs file I/O (tempfile, write, rename). The comment at line 97 tries to justify this, but it contradicts the safety principle at line 83.

**Impact**: In rare cases, if the signal fires during another `save_state()` call, the temp file from the in-progress write could conflict. The atomic rename pattern makes corruption unlikely, but it's not guaranteed safe.
